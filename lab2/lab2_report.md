## Ход работы

### 1. Создание сервиса с минимальными ресурсами
Для работы был выбран дефолтный контейнер `hello` из раздела Demo containers. При настройке были указаны минимальные ресурсы для экономии бюджета и выполнения условий лабы:
* **CPU:** 1
* **Memory:** 128 MiB
* **Autoscaling:** Max instances 1
* **Authentication:** Allow unauthenticated invocations

> ![Настройки ресурсов](сюда_вставь_свой_скриншот_с_128MiB)

### 2. Тестирование сервиса
После успешного развертывания сервис получил уникальный URL. При переходе по ссылке открывается приветственная страница "It's running!".

> **Скриншот №2: Работающий сервис по URL**
> ![Работающий сервис](твоя_ссылка_с_единорогом)

---

### 3. Анализ логов и метрик
В разделе **Observability** были проанализированы логи и метрики:
* **Logs:** Подтверждают запуск контейнера и прохождение HTTP-запросов (код 200).
* **Metrics:** Отображают всплеск активности (Request Count) в момент обращения к сервису.

---

### 4. Изменение конфигурации (Порт 8090)
Через функцию **Edit & Deploy New Revision** порт контейнера был изменен с 8080 на 8090. 
* В ходе эксперимента проверялась доступность сервиса после смены порта. 
* Была создана вторая ревизия `hellodk-00002-5rg`.

---

### 5. Распределение трафика
С помощью инструмента **Manage Traffic** и опции **Split traffic across multiple revisions**, нагрузка была распределена между двумя версиями сервиса в соотношении **50% на 50%**.

> **Скриншот №3: Распределение трафика 50/50**
> ![Распределение трафика](скриншот_где_две_ревизии_по_50_процентов)

---

### 6. Удаление ресурсов
После завершения исследований все созданные сервисы были удалены для предотвращения лишних затрат ресурсов облачной платформы.

---

## Вывод
В ходе лабораторной работы были изучены принципы работы Cloud Run, процесс развертывания контейнерных приложений, а также влияние параметров конфигурации (порт, трафик) на работоспособность сервиса. Работа с ревизиями позволила наглядно увидеть механизм плавного переключения трафика между версиями приложения.
